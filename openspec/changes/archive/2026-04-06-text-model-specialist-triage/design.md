## Context

Specialist tools (`specialist_billing`, `specialist_sales`, etc.) return a `response.create` payload with triage instructions. The Realtime voice model receives these instructions and generates audio — but it consistently ignores the multi-step triage flow (ask clarifying question → wait for answer → transfer). Instead, it transfers immediately.

The current flow:
1. Router dispatches to specialist (e.g., billing)
2. Specialist tool builds triage prompt with history → returns `response.create` dict
3. Coordinator emits filler voice (string prompt, parallel)
4. Coordinator emits specialist voice (dict prompt)
5. Bridge sends `response.create` to Realtime API
6. **Realtime model interprets instructions freely** → ignores triage rules

The bridge already handles two prompt types: `dict` (full response.create) and `str` (simple text wrapped in minimal response.create). Fillers use `str` prompts. Specialist responses use `dict` prompts with instructions.

## Goals / Non-Goals

**Goals:**
- Specialist triage responses are generated by a text model (`gpt-4o`) that reliably follows structured prompts
- The Realtime model vocalizes the text model's output literally, without reinterpretation
- Filler messages play during the text model call, masking latency
- Graceful fallback if the text model call fails (use current behavior as fallback)

**Non-Goals:**
- Changing the router prompt or routing logic
- Adding new specialist departments
- Streaming text model responses (batch is sufficient for 1-2 sentence triage responses)
- Frontend changes (this is backend-only)

## Decisions

### D1: Text model via OpenAI Chat Completions API

**Decision**: Use `httpx` async POST to OpenAI `/v1/chat/completions` with `gpt-4o`.

**Why over alternatives**:
- Already have `httpx` async in the project for LLM fallback routing (`llm_fallback_url`, `llm_fallback_api_key` in config)
- No need for a new SDK dependency — raw HTTP is consistent with existing LLM fallback code
- `gpt-4o` follows structured prompts reliably; `gpt-4o-mini` is also an option but `gpt-4o` is preferred for instruction-following quality

**Config**: Reuse existing `openai_api_key` from config. Add `specialist_model: str = "gpt-4o"` and `specialist_timeout_s: float = 5.0`.

### D2: Specialist tools return literal text, not response.create dict

**Decision**: Change specialist tool return type from `dict[str, Any]` (response.create payload) to `str` (literal text to speak).

**Why**: The bridge already handles `str` prompts by wrapping them in a minimal `response.create`. However, for literal vocalization we need the Realtime model to **speak the text exactly**, not interpret it. We'll use `response.create` with the text in a directive format: `"Say exactly the following to the customer: <text>"`.

**Coordinator change**: In `_on_model_router_action`, when `tool_result.ok`, the payload is now a `str`. Wrap it in a response.create with a "say exactly" instruction to prevent Realtime from paraphrasing.

### D3: Shared httpx client for specialist calls

**Decision**: Create a shared `httpx.AsyncClient` in the specialist tools module, initialized at startup via a `configure()` function called from `main.py` lifespan.

**Why**: Connection pooling across calls. Same pattern as the LLM fallback client. Timeout set per-request from config.

### D4: Filler flow unchanged

**Decision**: No changes to filler emission logic. The coordinator already emits fillers in parallel before awaiting the specialist tool. The only difference is that the tool now takes longer (~500-1500ms for text model call vs ~0ms for prompt construction), making fillers genuinely useful.

### D5: Fallback on text model failure

**Decision**: If the text model call fails (timeout, error, empty response), fall back to the current behavior — send the triage prompt as `instructions` in the response.create dict, letting the Realtime model interpret it. Log a warning.

**Why**: Better to have the Realtime model attempt triage (even if unreliable) than to fail silently or apologize.

## Risks / Trade-offs

- **[Latency]** Text model adds ~500-1500ms per specialist turn → **Mitigated** by filler messages playing in parallel. User hears "Let me connect you with billing" while text model processes.
- **[Cost]** ~1 additional API call per routed turn → Acceptable for MVP. Specialist routing is ~30-40% of turns.
- **[Double model dependency]** If OpenAI API is down, both routing and specialist triage fail → Same risk as today; no new failure mode.
- **[Vocalization fidelity]** Realtime model might paraphrase even with "say exactly" instruction → Mitigate with strong directive prompt. If still an issue, explore `conversation.item.create` + `response.create` (inject text as conversation item) as a future enhancement.
