## Context

The Coordinator currently constructs prompts as a single-turn exchange: `[system_base, system_policy, user_text]`. Each turn is independent — the voice agent has no knowledge of prior turns within the same call. This prevents natural multi-turn conversations where context carries forward (e.g., "tengo un problema con mi factura" → "¿cuánto debo?").

The conversation buffer must be:
- **Ephemeral**: Lives in memory per call, destroyed when the call ends (same lifecycle as `CoordinatorRuntimeState`)
- **Bounded**: Cannot grow unboundedly — prompts have token limits and latency budgets
- **Barge-in aware**: Cancelled turns must not pollute history

## Goals / Non-Goals

**Goals:**
- Accumulate completed turn context (user text + response type) within a single call
- Inject history into prompts so the voice agent has conversational continuity
- Bound history with a configurable sliding window (max turns) and token budget (max characters as proxy)
- Handle edge cases: barge-in (exclude cancelled turns), rapid successive turns, first turn (empty history)

**Non-Goals:**
- Cross-call memory or persistent conversation history (out of scope)
- Summarization or compression of history (future improvement)
- Modifying the Agent FSM or Router — history is injected at the prompt construction layer only
- Streaming or partial history updates during a turn

## Decisions

### Decision 1: ConversationBuffer as a standalone class

**Choice**: Create a new `ConversationBuffer` class in `backend/src/voice_runtime/conversation_buffer.py` rather than embedding history logic directly in the Coordinator.

**Rationale**: Separation of concerns. The buffer has its own logic (append, prune, format) that would clutter the Coordinator. It's also independently testable. The Coordinator instantiates it alongside `CoordinatorRuntimeState`.

**Alternative considered**: Store history as a list directly on `CoordinatorRuntimeState`. Rejected because state is a plain dataclass for tracking IDs/sets, not for complex logic like pruning and formatting.

### Decision 2: Turn entry structure

**Choice**: Each history entry is a lightweight dataclass:
```python
@dataclass(frozen=True)
class TurnEntry:
    seq: int
    user_text: str
    route_a_label: str       # e.g. "simple", "domain"
    policy_key: str | None    # e.g. "greeting", "guardrail_disallowed"
    specialist: str | None    # e.g. "billing" (only for domain routes)
```

**Rationale**: Store the classification result, not the full prompt or agent response. This keeps entries small and avoids duplicating policy text. The voice agent can infer from the classification what the system said (e.g., route_a=disallowed means a guardrail response was given).

**Alternative considered**: Store full assistant response text. Rejected because we don't have the actual voice output text (it's generated by the Realtime provider, not returned to us). Storing the policy key is the closest proxy.

### Decision 3: Sliding window with character budget

**Choice**: Two bounds — `max_history_turns` (default: 10) and `max_history_chars` (default: 2000). When either limit is exceeded, oldest entries are dropped first.

**Rationale**: Token counting requires a tokenizer which adds latency and complexity. Character count is a reasonable proxy (1 token ≈ 4 chars for Spanish/English). A 2000-char budget leaves ample room within typical 4096-8192 token context windows after system prompts.

**Alternative considered**: Exact token counting via tiktoken. Rejected for MVP — adds a dependency and ~1ms per call. Can be upgraded later without changing the interface.

### Decision 4: History format in prompt

**Choice**: Inject history as alternating user/assistant messages between the system messages and the current user message:

```python
[
    {"role": "system", "content": BASE_SYSTEM},
    {"role": "system", "content": POLICY_INSTRUCTIONS},
    # --- history starts ---
    {"role": "user", "content": "tengo un problema con mi factura"},
    {"role": "assistant", "content": "[billing] Processed as billing inquiry"},
    {"role": "user", "content": "¿cuánto debo?"},
    {"role": "assistant", "content": "[billing] Processed as billing inquiry"},
    # --- history ends ---
    {"role": "user", "content": "y desde cuándo?"},  # current turn
]
```

**Rationale**: The chat-message format is native to LLM APIs. Using `role: "assistant"` for prior responses gives the model clear conversational structure. The assistant content is a synthetic summary (not actual voice output) containing the route and specialist.

**Alternative considered**: Single `role: "system"` block with formatted history text. Rejected because chat-message format better leverages the model's conversation understanding.

### Decision 5: Barge-in and cancelled turns

**Choice**: Only finalized turns with a completed voice generation are added to history. Cancelled turns (barge-in, rapid successive) are never appended.

**Rationale**: A cancelled turn means the system response was interrupted — including it in history would create misleading context. The buffer's `append()` is called only after successful prompt construction and voice start emission.

### Decision 6: Configuration via Settings

**Choice**: Add `max_history_turns: int = 10` and `max_history_chars: int = 2000` to the existing `Settings` class in `backend/src/config.py`.

**Rationale**: Consistent with existing configuration pattern. Overridable via environment variables.

## Risks / Trade-offs

- **[Prompt length]** → History increases prompt size, adding latency to voice generation. Mitigation: Character budget cap (2000 chars) and sliding window ensure bounded growth. Monitor via existing `voice_turn_latency_ms` metric.

- **[Synthetic assistant messages]** → The assistant content is a classification summary, not actual voice output. The model might be confused by the format. Mitigation: Use a consistent, clear format like `[billing] Processed as billing inquiry`. Can iterate on the format based on voice quality testing.

- **[No persistence]** → History is lost if the process crashes mid-call. Mitigation: Acceptable for MVP — calls are typically short (< 5 minutes). Persistence can be added later by serializing the buffer to Redis.

- **[Character vs token budget]** → Character counting is approximate. Mitigation: Use a conservative ratio (2000 chars ≈ 500 tokens), well within budget. Upgrade to tiktoken later if needed.

## Open Questions

- Should the assistant summary format include the actual policy instructions snippet, or just the label? Starting with label-only for simplicity.
- Should specialist tool results be included in history? Starting with no — only the route/policy is recorded. Tool results can be added in a future iteration.
